## 1. QUÉ ES GATEWAY API Y POR QUÉ EXISTE

Gateway API es el proyecto oficial de Kubernetes para L4/L7 routing. No es una versión 2 de Ingress — es un rediseño completo que resuelve los problemas estructurales de Ingress que el espectador ya conoce del vídeo anterior.

**En una frase:** Gateway API es lo que Ingress debería haber sido desde el principio.

**Estado actual:**
- Proyecto bajo SIG-Network (kubernetes-sigs)
- API Core (GatewayClass, Gateway, HTTPRoute) en **GA desde v1.0 (Octubre 2023)**
- Versión actual: **v1.5.1**
- Soportado por todas las implementaciones principales de Ingress

**Lo que Ingress NO puede hacer sin anotaciones propietarias** (repaso breve, asume conocimiento previo):
- Routing por headers
- Traffic splitting/weighted routing
- Redirects y URL rewrites en spec estándar
- Routing TCP/UDP
- Separación de responsabilidades entre infra y aplicación
- Cross-namespace routing

---

## 2. LOS TRES ROLES (ROLE-ORIENTED DESIGN)

Este es el cambio conceptual más importante respecto a Ingress. Gateway API está diseñado explícitamente alrededor de tres roles que corresponden a personas reales en una organización:

### Ian — Infrastructure Provider
Gestiona la **infraestructura** que permite tener múltiples clusters para múltiples tenants. Se preocupa de todos los clusters colectivamente.
→ **Recurso:** `GatewayClass`

### Chihiro — Cluster Operator
Gestiona **un cluster** concreto. Asegura que cumple los requisitos del negocio, decide qué infraestructura de networking está disponible para los developers.
→ **Recurso:** `Gateway`

### Ana — Application Developer
Desarrolla aplicaciones en el cluster. No le importa cómo funciona el LB subyacente — solo quiere exponer sus services.
→ **Recurso:** `HTTPRoute`, `GRPCRoute`, `TCPRoute`, etc.

**Por qué esto importa en la práctica:** en Ingress, un developer puede crear un Ingress que afecta a la configuración del load balancer de todo el cluster (anotaciones de nginx que tocan el controlador global). En Gateway API, el Cluster Operator define los Gateways con las políticas que corresponden, y el Developer solo puede crear Routes que se adjunten a esos Gateways — sin acceso a la configuración de infraestructura.

## 3. MODELO DE OBJETOS — LOS RECURSOS

### GatewayClass

**Ámbito:** cluster-scoped (no tiene namespace)
**API version:** `gateway.networking.k8s.io/v1` (GA)
**Análogo:** `StorageClass` para PersistentVolumes, `IngressClass` para Ingress

Define un *tipo* de gateway — es decir, qué controlador lo va a gestionar. Un cluster puede tener múltiples GatewayClasses (ej. una para tráfico externo con Envoy, otra para tráfico interno con Nginx).

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: nginx
spec:
  controllerName: gateway.nginx.org/nginx-gateway-controller
```

El `controllerName` es el identificador del controlador que reconcilia los Gateways de esta clase. Cada implementación publica su GatewayClass al instalarse.

### Gateway

**Ámbito:** namespaced (generalmente en namespace de infraestructura o kube-system)
**API version:** `gateway.networking.k8s.io/v1` (GA)

Describe un punto de acceso al cluster — piensa en él como la definición del load balancer. Contiene:

- **`gatewayClassName`**: referencia a la GatewayClass
- **`listeners`**: cómo escuchar tráfico — puerto, protocolo, hostname, TLS

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: prod-gateway
  namespace: infra
spec:
  gatewayClassName: nginx
  listeners:
  - name: http
    protocol: HTTP
    port: 80
  - name: https
    protocol: HTTPS
    port: 443
    hostname: "*.example.com"
    tls:
      mode: Terminate
      certificateRefs:
      - name: wildcard-tls-secret
        namespace: infra
```

**Conceptos clave de Listeners:**
- Cada Listener es independiente: puerto + protocolo + (opcionalmente) hostname
- Los Listeners deben ser *distintos* — no pueden tener conflictos. Si hay conflicto, el Gateway entero queda inválido (no pasa a `Accepted`)
- El controlador puede añadir configuración implícita (ej. IPs de los LBs) que no estaba en el spec original

### HTTPRoute

**Ámbito:** namespaced (en el namespace de la aplicación)
**API version:** `gateway.networking.k8s.io/v1` (GA)

Define las reglas de routing para tráfico HTTP/HTTPS. Se adjunta a uno o varios Listeners de un Gateway via `parentRefs`.

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: myapp-route
  namespace: myapp
spec:
  parentRefs:
  - name: prod-gateway
    namespace: infra          # cross-namespace reference
    sectionName: https        # qué Listener del Gateway
  hostnames:
  - "myapp.example.com"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /api
      headers:
      - name: X-Version
        value: v2
    backendRefs:
    - name: myapp-v2-service
      port: 8080
      weight: 90
    - name: myapp-v1-service
      port: 8080
      weight: 10
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: myapp-service
      port: 8080
    filters:
    - type: RequestHeaderModifier
      requestHeaderModifier:
        add:
        - name: X-Request-Id
          value: "my-app"  # ⚠️ CORE SPEC: value admite solo strings literales.
          # La sintaxis "%REQ(x-request-id)%" es específica de Envoy Proxy, NO de Gateway API core.
```

**Lo que HTTPRoute puede hacer que Ingress no puede (en spec estándar):**
- Matching por headers (`headers[]`)
- Matching por query parameters (`queryParams[]`)
- Matching por método HTTP (`method`)
- Traffic splitting con pesos (`backendRefs[].weight`)
- Redirects (`RequestRedirect` filter)
- URL rewrites (`URLRewrite` filter)
- Header modification (add, set, remove)
- Request mirroring (`RequestMirror` filter)

### GRPCRoute

**Ámbito:** namespaced
**API version:** `gateway.networking.k8s.io/v1` (GA desde v1.1)

Routing de tráfico gRPC — permite matching por service name y method name, algo que HTTPRoute no puede hacer de forma semántica.

### TLSRoute

**Ámbito:** namespaced
**API version:** `gateway.networking.k8s.io/v1` (GA desde v1.5)

Routing de TLS por SNI. Para casos donde quieres hacer routing basado en el nombre del servidor TLS pero sin terminar la conexión (TLS passthrough).

### TCPRoute y UDPRoute

**API version:** `gateway.networking.k8s.io/v1alpha2` (Experimental — alpha)

Routing TCP y UDP puro — sin inspección de L7. Útil para bases de datos, protocolos binarios, DNS.

### ReferenceGrant

**Ámbito:** namespaced
**API version:** `gateway.networking.k8s.io/v1beta1` (Standard channel)

Controla el acceso cross-namespace. Cuando un HTTPRoute en namespace `myapp` quiere adjuntarse a un Gateway en namespace `infra`, el Cluster Operator debe crear un `ReferenceGrant` en `infra` autorizando ese acceso.

```yaml
apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: allow-myapp-routes
  namespace: infra           # namespace del recurso referenciado
spec:
  from:
  - group: gateway.networking.k8s.io
    kind: HTTPRoute
    namespace: myapp          # namespace autorizado a referenciar
  to:
  - group: gateway.networking.k8s.io
    kind: Gateway
```

## 4. COMPARATIVA INGRESS VS GATEWAY API

| Aspecto                 | Ingress                                       | Gateway API                                          |
| ----------------------- | --------------------------------------------- | ---------------------------------------------------- |
| **Separación de roles** | ❌ Ninguna — developer puede afectar LB config | ✅ GatewayClass (infra) / Gateway (ops) / Route (dev) |
| **Header routing**      | Solo via anotaciones propietarias             | ✅ Nativo en HTTPRoute spec                           |
| **Traffic splitting**   | Solo via anotaciones propietarias             | ✅ Nativo con `weight` en backendRefs                 |
| **Redirects/Rewrites**  | Anotaciones nginx/traefik etc.                | ✅ Filters en HTTPRoute                               |
| **TCP/UDP routing**     | ❌ No soportado                                | ✅ TCPRoute/UDPRoute (experimental)                   |
| **gRPC routing**        | Hacky vía anotaciones                         | ✅ GRPCRoute nativo                                   |
| **Cross-namespace**     | ❌ No — Service y Ingress mismo namespace      | ✅ Via ReferenceGrant                                 |
| **Service Mesh**        | ❌ No                                          | ✅ GAMMA initiative (GA desde v1.1)                   |
| **Portabilidad**        | ❌ Anotaciones distintas por controlador       | ✅ Spec estándar en todos los controllers             |
| **TLS**                 | Configurado en Ingress spec                   | ✅ En Gateway Listener (infra concern)                |

**La diferencia clave en términos de portabilidad:** un HTTPRoute con traffic splitting o header routing funciona igual en Nginx Gateway Fabric, Envoy Gateway, Cilium, o Traefik. En Ingress, necesitas anotaciones distintas para cada uno.

## 5. ESTADO DE LAS APIS (CANALES)

Gateway API usa dos release channels:

**Standard Channel** (usar en producción):
| Recurso | API Version | Canal | Desde versión |
|---|---|---|---|
| GatewayClass | v1 | Standard | v0.5.0 |
| Gateway | v1 | Standard | v0.5.0 |
| HTTPRoute | v1 | Standard | v0.5.0 |
| GRPCRoute | v1 | Standard | v1.1.0 |
| TLSRoute | v1 | Standard | v1.5.0 |
| ReferenceGrant | v1beta1 | Standard | — |

**Experimental Channel** (features nuevas, sin garantías de compatibilidad):
| Recurso | API Version | Estado |
|---|---|---|
| TCPRoute | v1alpha2 | Experimental |
| UDPRoute | v1alpha2 | Experimental |
| BackendTLSPolicy | v1alpha3 | Experimental |
| ListenerSet | v1alpha2 | Experimental |

**Regla práctica:** instala siempre el Standard Channel en producción. El Experimental Channel solo si necesitas una feature específica de alpha.

```bash
# Instalar Standard Channel (CRDs)
kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.5.1/standard-install.yaml

# Instalar Experimental Channel (todo: Standard + Experimental)
kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.5.1/experimental-install.yaml
```

### Completamente Conformantes (Conformant)

| Implementación | Notas | Estado |
|---|---|---|
| **Cilium** | CNI con gateway API nativo | Conformant |
| **Envoy Gateway** | CNCF, referencia oficial, basado en Envoy | Conformant (GA) |
| **Istio** | Service mesh + ingress gateway | Conformant (GA) |
| **kgateway** (ex-Gloo Gateway OSS) | Solo Foundation, CNCF sandbox | Conformant (GA) |
| **NGINX Gateway Fabric** | Sucesor de ingress-nginx para Gateway API | Conformant (GA) |
| **Traefik Proxy** | Muy popular en homelabs y SMB | Conformant (GA) |

### Parcialmente Conformantes (Partially Conformant)

| Implementación | Notas |
|---|---|
| AWS Load Balancer Controller | Para clusters EKS |
| Azure Application Gateway for Containers | AKS |
| Contour (VMware) | |
| GKE Gateway | Google Kubernetes Engine |
| Kong Ingress Controller | |

**Para la audiencia del curso (que viene de ingress-nginx):**
La ruta natural es **NGINX Gateway Fabric** — mismo fabricante (NGINX Inc.), diseñado específicamente como sucesor de ingress-nginx. Soporta el mismo modelo de Ingress pero con Gateway API como API principal.

### Integraciones (no son controllers, trabajan con Gateway API)

- **cert-manager** (alpha) — gestión de certificados TLS para Gateway objects
- **Flagger** (public preview) — progressive delivery / canary deployments
- **Argo Rollouts** (alpha) — progressive delivery integration
- **Kuadrant** (GA) — API management y rate limiting policies

## 7. EL MODELO DE ATTACH — CÓMO SE CONECTAN LOS OBJETOS

Este es el concepto que más confunde a quien viene de Ingress.

En Ingress, todo está en un solo objeto. En Gateway API, hay tres objetos separados que se "attach" unos a otros:

```
GatewayClass          Gateway                    HTTPRoute
(cluster-scoped)  ←── (namespace: infra)    ←── (namespace: myapp)
"nginx"               listeners:                 parentRefs:
                      - port: 443                - name: prod-gateway
                        protocol: HTTPS            namespace: infra
                        hostname: *.example.com    sectionName: https
                                                 hostnames:
                                                 - myapp.example.com
```

**Reglas de adjunción:**
1. El HTTPRoute especifica en `parentRefs` a qué Gateway (y opcionalmente, qué Listener) se adjunta
2. El Gateway puede restringir qué Routes puede aceptar via `allowedRoutes` en cada Listener
3. Si Gateway y HTTPRoute están en namespaces distintos, necesitas un `ReferenceGrant`

**Status — cómo verificar que todo está conectado:**
```bash
# Ver si el Gateway está aceptado por el controlador
kubectl describe gateway prod-gateway -n infra
# Buscar: status.conditions con type=Accepted, status=True

# Ver si el HTTPRoute está adjunto al Gateway
kubectl describe httproute myapp-route -n myapp
# Buscar: status.parents[].conditions con type=Accepted, status=True
```

## 8. EJEMPLO COMPLETO — DE INGRESS A GATEWAY API

### Antes (Ingress)
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "20"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - myapp.example.com
    secretName: myapp-tls
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: myapp-service
            port:
              number: 8080
```

### Después (Gateway API)
```yaml
# Gateway (gestionado por Cluster Operator)
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: prod-gateway
  namespace: infra
spec:
  gatewayClassName: nginx
  listeners:
  - name: https
    protocol: HTTPS
    port: 443
    hostname: "*.example.com"
    tls:
      mode: Terminate
      certificateRefs:
      - name: wildcard-tls-secret
---
# HTTPRoute (gestionado por Application Developer)
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: myapp-route
  namespace: myapp
spec:
  parentRefs:
  - name: prod-gateway
    namespace: infra
    sectionName: https
  hostnames:
  - "myapp.example.com"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: myapp-v2-service
      port: 8080
      weight: 20
    - name: myapp-service
      port: 8080
      weight: 80
```

La diferencia estructural: el TLS ya no es responsabilidad del developer — el Cluster Operator lo configura en el Gateway. El developer solo define cómo se enruta el tráfico.
