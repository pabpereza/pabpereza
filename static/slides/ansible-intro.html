¿Qué es Ansible?

Plataforma open-source · Red Hat · 2012

⚙️ Gestión de configuración

100 servidores web con la misma config de Nginx, mismos certs SSL y políticas de seguridad.

🚀 Despliegue de aplicaciones

Integra con Jenkins, GitLab CI o GitHub Actions para deployments automáticos.

☁️ Gestión de la nube

Aprovisiona recursos en AWS, Azure, GCP y más.

🎼 Orquestación

Coordina bases de datos, balanceadores y servidores de app en el orden correcto.

🛡️ Cumplimiento

Audita y aplica PCI-DSS, HIPAA, SOC2 de forma automática y auditable.

🔄 Disaster Recovery

Reconstruye toda tu infra en minutos. El desastre se convierte en inconveniente.

Imperativo vs Declarativo

El cambio de mentalidad más importante

❌ Bash — Imperativo 
# Si ejecutas esto 2 veces → puede fallar
apt-get install -y nginx
service nginx start
echo "server_name myapp;" >> /etc/nginx.conf

Problema: no idempotente. Estado final impredecible.

✓ Ansible — Declarativo 
- name: Asegurar Nginx instalado
  apt:
    name: nginx
    state: present
- name: Asegurar Nginx corriendo
  service:
    name: nginx
    state: started
    enabled: yes

100 ejecuciones, mismo resultado.

🔁 Idempotencia: mismo resultado, ejecutes el playbook una o cien veces. Si Nginx ya está, Ansible no toca nada.

Push vs Pull

Dos filosofías distintas para gestionar cambios

🚀 Push (Ansible)

  • Tú decides cuándo se ejecutan los cambios
  • Control total del timing
  • Ideal para CI/CD y deployments bajo demanda
  • Cambios inmediatos y visibles
📌 Mejor para: despliegues puntuales, cambios críticos, pipelines CI/CD

🔄 Pull (Chef/Puppet)

  • Los agentes consultan periódicamente al master
  • Cambios eventuales (cada ~30 min)
  • Puede autocurarse sin intervención
  • Ideal para infraestructura masiva
📌 Mejor para: infra masiva que debe autocorregirse continuamente

Flujo de ejecución

Lo que pasa cuando lanzas ansible-playbook site.yml

1. Usuario
→
Ejecuta ansible-playbook site.yml
2. Ansible CLI
→
Lee el inventario y resuelve la lista de hosts
3. Compilación
→
Compila los módulos Python necesarios
4. SSH
→
Transfiere el módulo al nodo gestionado
5. Ejecución
→
El nodo ejecuta el módulo localmente
6. Resultado
→
Devuelve JSON con el resultado (ok / changed / failed)
7. Limpieza
→
Elimina el módulo temporal del nodo
