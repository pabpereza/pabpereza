# CoreDNS, servicios y DNS en Kubernetes

Cuando en los ejemplos del curso un pod se conecta a `mydb` o a `servicio-app1`, hay un mecanismo silencioso haciendo magia: el DNS interno del cluster. Entenderlo es clave para desarrollar aplicaciones (y para la parte de troubleshooting de cualquier examen), porque cuando el DNS falla, **todo** falla.

## CoreDNS: el servidor DNS del cluster
**CoreDNS** es el servidor DNS que kubeadm despliega por defecto en todos los clusters. Corre como un deployment en `kube-system`, expuesto por un service llamado `kube-dns` (el nombre se mantiene por compatibilidad histórica):

```bash
kubectl -n kube-system get deployment coredns
kubectl -n kube-system get svc kube-dns
```

Cada pod del cluster se configura automáticamente para usarlo: el kubelet escribe en el `/etc/resolv.conf` de cada contenedor la IP del service `kube-dns` y los dominios de búsqueda:

```bash
kubectl exec -it mi-pod -- cat /etc/resolv.conf
# nameserver 10.96.0.10
# search default.svc.cluster.local svc.cluster.local cluster.local
# options ndots:5
```

Esa línea `search` es la que permite usar nombres cortos, como veremos ahora.

## Nombres DNS de los services
Todo service recibe automáticamente un nombre DNS con esta estructura:

```text
<servicio>.<namespace>.svc.cluster.local
```

Gracias a los dominios de búsqueda, desde un pod puedes referirte a un service de cuatro formas, de más corta a más explícita:

```bash
curl http://mydb                                  # Mismo namespace
curl http://mydb.production                       # Otro namespace
curl http://mydb.production.svc                   # Más explícito
curl http://mydb.production.svc.cluster.local     # FQDN completo
```

La regla práctica: **dentro del mismo namespace basta el nombre corto; entre namespaces, añade el namespace**. Este detalle cae constantemente en los exámenes ("¿por qué la app del namespace `frontend` no resuelve `mydb`?" — porque mydb vive en otro namespace).

Lo que resuelve ese nombre es la **ClusterIP** del service. A partir de ahí, kube-proxy hace el resto, como vimos en [services](./111.Services.md).

## DNS de los headless services
Los services *headless* (`clusterIP: None`), que conocimos con los [StatefulSets](./109.DSySS.md), no tienen ClusterIP: su nombre DNS devuelve directamente **las IPs de todos los pods**:

```bash
nslookup mongo.default.svc.cluster.local
# Devuelve las IPs de mongo-0, mongo-1 y mongo-2
```

Y cada pod del StatefulSet recibe además su propio registro estable:

```text
<pod>.<servicio>.<namespace>.svc.cluster.local
# mongo-0.mongo.default.svc.cluster.local
```

Esto es lo que permite a las bases de datos distribuidas configurar la replicación entre miembros concretos, algo imposible con un service normal que balancea aleatoriamente.

## Políticas DNS del pod
El comportamiento DNS de cada pod se controla con `dnsPolicy` en su spec:

- `ClusterFirst` (por defecto): resuelve primero contra CoreDNS; lo que no sea del cluster se reenvía a los DNS externos.
- `Default`: hereda el `/etc/resolv.conf` del nodo (ojo: ¡no es el valor por defecto, pese al nombre!).
- `None`: configuración manual completa mediante `dnsConfig`.
- `ClusterFirstWithHostNet`: para pods con `hostNetwork: true` que aun así quieran usar el DNS del cluster.

```yaml
spec:
  dnsPolicy: "None"
  dnsConfig:
    nameservers:
    - 1.1.1.1
    searches:
    - mi-dominio.interno
    options:
    - name: ndots
      value: "2"
```

## El Corefile: configurar CoreDNS
La configuración de CoreDNS vive en un ConfigMap llamado `coredns` en `kube-system`, con su propio formato llamado **Corefile**:

```bash
kubectl -n kube-system get configmap coredns -o yaml
```

```text
.:53 {
    errors
    health
    kubernetes cluster.local in-addr.arpa ip6.arpa {
        pods insecure
        fallthrough in-addr.arpa ip6.arpa
    }
    prometheus :9153
    forward . /etc/resolv.conf
    cache 30
    loop
    reload
    loadbalance
}
```

Los plugins que más nos interesan: `kubernetes` (genera los registros de services y pods), `forward` (a dónde se reenvían las consultas externas) y `cache`. Un caso real típico es añadir un bloque para reenviar un dominio corporativo a un DNS interno:

```text
empresa.local:53 {
    forward . 10.0.0.53
}
```

Tras editar el ConfigMap, el plugin `reload` aplica los cambios en un par de minutos (o reinicia el deployment con `kubectl -n kube-system rollout restart deployment coredns`).

## Troubleshooting de DNS
La caja de herramientas para "no resuelve":

1. **Probar la resolución desde un pod de pruebas**:
```bash
kubectl run dns-test --image=busybox:1.36 --rm -it --restart=Never -- nslookup kubernetes.default
```
Si esto funciona, el DNS del cluster está vivo y el problema es del nombre que usas (¿namespace correcto? ¿service existe?).

2. **Comprobar CoreDNS y su service**:
```bash
kubectl -n kube-system get pods -l k8s-app=kube-dns
kubectl -n kube-system logs -l k8s-app=kube-dns
kubectl -n kube-system get endpoints kube-dns   # ¿Tiene endpoints?
```

3. **Comprobar que el service destino tiene endpoints** (la causa más común: el DNS resuelve bien, pero el service no tiene pods detrás por un selector mal escrito):
```bash
kubectl get endpoints mydb
```

4. **Revisar el resolv.conf del pod afectado** por si su `dnsPolicy` no es la esperada.

## Resumen
- CoreDNS sirve el DNS del cluster; cada pod lo usa vía el service `kube-dns` inyectado en su `/etc/resolv.conf`.
- Formato de nombres: `servicio.namespace.svc.cluster.local`; nombre corto dentro del namespace, cualificado entre namespaces.
- Los headless services devuelven las IPs de los pods, y los StatefulSets dan un registro DNS estable a cada pod.
- `dnsPolicy`/`dnsConfig` personalizan la resolución por pod; el Corefile la personaliza a nivel de cluster.
- Ante fallos: pod de pruebas con `nslookup`, estado de CoreDNS y, sobre todo, los endpoints del service.
